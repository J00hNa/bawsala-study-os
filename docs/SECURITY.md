# Security Model

## ما الذي نحميه؟

- بيانات الدراسة الخاصة بكل مستخدم.
- جلسات Auth في المتصفح.
- سلامة المزامنة عند وجود أكثر من جهاز.
- مفتاح الإدارة `service_role`.
- التطبيق من JSON ضخم أو حقول خبيثة مستوردة.

## حدود الثقة

### داخل المتصفح

- `config.js` يحتوي قيمًا عامة فقط.
- Access/refresh tokens تُحفظ في `localStorage` لتوفير جلسة PWA بسيطة. لذلك منع XSS أساسي؛ أي XSS ناجح يمكنه سرقة الجلسة.
- CSP يمنع inline scripts والمصادر غير المصرّح بها. بقي `style-src 'unsafe-inline'` لأن التطبيق يولّد بعض الأنماط ديناميكيًا؛ هذه مخاطرة متبقية ويجب إزالتها مستقبلًا بإعادة هيكلة الواجهة.
- Service Worker لا يعترض ولا يخزّن أي طلب cross-origin، لذلك لا تدخل استجابات Auth أو قاعدة البيانات إلى Cache Storage.

### داخل قاعدة البيانات

- RLS مفعّل و`FORCE ROW LEVEL SECURITY` على جدولي الحالة والنسخ.
- كل سياسة تقارن `auth.uid()` مع `user_id`.
- الواجهة تملك SELECT لصفها فقط.
- الكتابة تتم عبر `sync_study_state` مع `expected_revision`، وليس UPDATE مباشرًا؛ هذا يمنع lost updates الصامتة.
- حجم الحالة محدود بـ 1.5 MB ونوعها يجب أن يكون JSON object.
- النسخ الاحتياطية تُنشأ قبل التحديث بحد زمني، ويُحتفظ بآخر 8 نسخ.

### داخل Edge Function

- `service_role` موجود فقط كمتغير بيئة خادمي.
- الدالة تتحقق من Bearer session عبر Auth قبل حذف المستخدم.
- الحذف يطبق فقط على `user.id` المستخرج من الجلسة؛ لا يقبل user ID من الطلب.
- CORS يفشل مغلقًا إذا لم يُضبط `ALLOWED_ORIGINS`، ويرفض origin غير المدرج.

## الحماية من إساءة الاستخدام

- Supabase Auth rate limits.
- Turnstile اختياري ومتكامل مع طلبات signup/signin/recovery.
- كلمات مرور لا تقل عن 10 أحرف في الواجهة، ويجب تكرار السياسة خادميًا في Supabase.
- تأكيد البريد موصى به قبل الإطلاق العام.
- رسائل الخطأ لا تعرض service keys أو تفاصيل قاعدة البيانات.

## استيراد البيانات

`normalizeState()` لا يثق بملف JSON:

- يحدد أنواع الحقول والقيم المسموحة.
- يقيد الأطوال والأعداد وأحجام القوائم.
- يقبل روابط `http` و`https` فقط.
- يعيد توليد/تنظيف المعرّفات غير الآمنة.
- يرفض التواريخ غير الحقيقية مثل `2026-02-31`.

يوجد حد آخر داخل Postgres لحجم الوثيقة. الحماية على الواجهة وحدها ليست كافية؛ لذلك توجد الحماية في الطبقتين.

## أشياء غير موجودة ولا يجوز الادعاء بوجودها

- لا يوجد end-to-end encryption.
- لا يوجد MFA UI حاليًا.
- لا يوجد Admin dashboard أو صلاحيات أدوار متعددة.
- لا يوجد WAF مخصص أو SIEM.
- لا يوجد ضمان uptime على الخطط المجانية.
- النسخ المحلية ليست بديلًا عن backup مؤسسي خارج المزود.

## قائمة إلزامية قبل إطلاق عام

- [ ] غيّرت Site URL وRedirect URLs من localhost إلى رابط الإنتاج.
- [ ] فعّلت Email confirmation.
- [ ] ربطت Custom SMTP واختبرت التأكيد والاستعادة.
- [ ] فعّلت Turnstile وقيّدت hostnames.
- [ ] وضعت `ALLOWED_ORIGINS` الدقيقة لدالة حذف الحساب.
- [ ] تأكدت أن Git history لا يحتوي `service_role` أو SMTP secret أو Turnstile secret.
- [ ] شغّلت Security Advisor وراجعت RLS.
- [ ] اختبرت عزل مستخدمين مختلفين يدويًا.
- [ ] اختبرت التعارض، offline، والاستعادة.
- [ ] أعددت تصدير/نسخًا دوريًا خارج Supabase إذا أصبحت البيانات مهمة.

## الإبلاغ عن مشكلة أمنية

لا تنشر مفاتيح أو بيانات مستخدم في Issue عام. احتفظ بالـ request IDs ووقت الخطأ وشارك أقل قدر من البيانات اللازمة عبر قناة خاصة للمشروع.
